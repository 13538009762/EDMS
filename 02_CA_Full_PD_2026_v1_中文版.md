## 引言

### 1.1 职业能力名称与描述：产品开发

### 1.2 职业能力描述

「产品开发」能力旨在培养创建与支持面向业务的软件产品所需的综合能力。该方向要求参赛者在知识技能上覆盖软件开发生命周期的各个阶段。

在该能力中，重点不仅在于技术栈，更在于开发过程本身，以及比较不同软件产品构建方法并依据评估策略选择最切合方案的原则。

该能力通过参赛者在开发团队中的自主组织、完成既定业务任务以及与其他能力参赛者交流经验，来激发参赛者的潜能。

该能力允许使用多种开发栈与编程语言，但**不得**使用低代码/无代码类方案：如 1C、SAP 及同属该类别的其他方案。

参赛者仅可使用具有开放许可（MIT、GPL 等）的公开开源库与框架。成果须在开放许可下发布，因此**禁止**使用单位内部封闭的自有代码。团队拟使用的开发工具、库、框架及运行环境须在团队的「工具箱」中声明；例外情况仅可由授权专家共同决定。

### 1.3 适用范围

每位专家与参赛者均有义务熟悉本竞赛任务书。

### 1.4 配套文件

本竞赛任务书仅包含与对应职业能力相关的信息，须与下列文件**一并**使用：

- 《锦标赛举办条例》
- 《技术说明：产品开发》
- 《锦标赛参赛者劳动安全须知》

根据 AtomSkills 行业锦标赛中「面向业务的软件解决方案」能力的试测与参赛成绩（2022–2025 年统计），确定如下职业水平门槛：

- **职业联赛**：在「产品开发」能力中确认职业身份的门槛——**至少 60 分**
- **学生联赛**：在「产品开发」能力中确认职业身份的门槛——**至少 45 分**

## 竞赛参与形式

产品开发为**团队能力**。每队由 **3 名**参赛者组成。每队须由专家导师或专家开发者代表。一名专家最多可代表**同一部门**的 2 支队伍；学生联赛中，可为**同一教育机构**的 2 支队伍。

### 模块 A：按功能要求完成任务

#### 总体描述

企业同时处理大量内部文件：规章、说明、合同、公务函件及项目文档等。在编制与审批过程中，多名员工参与，从而产生多轮修订、批注以及由用户以不同方式保存的多个版本。若缺乏统一的编辑与变更管控环境，将难以厘清**何人、何时、做了何种修改**，以及**哪一版为当前有效版本**。

公司发布软件开发招标，目的是依据参赛者对功能能力的展示与最终方案陈述，遴选最优技术路线。

为优化文档工作，拟建设**电子文档管理信息系统**，核心任务是：**集中**完成文档的创建、编辑与审批，并记录变更历史、管理版本、支持用户协同。

产品面向经常处理内部文档的企业员工，包括对口部门专员、管理人员、项目组成员以及负责文稿编制与审批的行政人员。对管理者尤为重要：常需将同一份文稿发往多个部门、汇总各方修订、将各条编辑分支合并为经批准的单一版本。若无集中方案，文档副本易丢失，内容分散在各参与人手中。结果是，在历时数月、伴随大量会议讨论的长期文档工作中，最初需求与已约定功能可能部分失准或走样。系统实施后应能维护信息完整性、保证全部变更可追溯，并为管理层提供工具，使每份文档呈现**单一且最新**的全貌。

#### 输入数据

系统使用企业**主数据（参考数据）**，运行中必须加以利用。数据包括：公司员工列表、管理人员、职务、部门等。这些主数据用于用户识别、权限分配及审批流程组织。

输入数据格式为**含多张工作表的电子表格**。应用须提供模块，通过上传 **\*.XLSX** 文件导入当前主数据，并**在导入前清除**此前已上传的信息。

并非所有输入字段都必须处理：部分字段可作为元数据，是否在系统中保存由开发者自行决定。

#### 总体系统要求

- 登录仅允许使用输入数据中指定的**登录名**；**不要求**输入密码。

#### 个人文档库

授权后，用户可访问**个人文档库**。库中文档归属特定用户，可用于发起审批流程及向其他员工授权以协同工作。

#### 文档创建与编辑

- 文档可在系统内直接创建，也可通过导入 **DOCX** 上传。导入文档或从剪贴板粘贴时，须**保留**原稿结构与内容。
- 系统须内置**文本编辑器**，支持编辑与排版。编辑器至少须支持：
  - 字体族与字号选择
  - 文本样式（粗体、斜体、下划线、删除线）
  - 多级标题样式
  - 文字与背景颜色
  - 对齐方式
  - 行距与段落缩进
  - 项目符号与编号列表
- 编辑器须提供全文**查找与替换**。系统还须支持**标点检查**。
- 文档须具备**页面设置**（方向、页边距、纸张规格）。须支持在**右下角**自动编排页码。
- 系统须支持导出为 **DOCX** 与 **PDF**；用户可生成所选格式文件并下载到本地。
- 关闭或退出文档时，系统须**自动保存**全部更改，**无需**用户再次确认。

#### 文档批注

- 系统须实现**批注模块**，用户可对文档中选中文本片段添加批注。
- 文档所有者、获授权的员工以及审批流程中的全体参与者均可留言。批注作者可删除自己的批注；所有者与获权员工可将批注标为**已完成**。
- 每条批注须包含：作者、创建日期与时间、状态（进行中 / 已完成）。系统须支持以**讨论串**形式回复批注。
- 须提供全部批注列表，并可按作者、状态、日期筛选。
- 批注须在视觉上与对应文本片段关联并在编辑器中显示。当所附文本发生变化时，系统须保持批注与当前片段的关联，或**提示**用户需要更新关联。
- 须支持**隐藏**已完成批注，而不从文档历史中删除。

#### 协同编辑

- 系统须支持多名用户对**同一文档**协同工作。文档所有者可向其他员工授予不同权限：**查看、编辑、批注**。授权时须遵循合理逻辑。
- 协同过程中所有用户作出的变更须**交互式**展示，并标明**修改人**。
- 在实时协同模式下，系统须显示所有正在编辑用户的**活动光标**；光标须以颜色区分，并显示每名用户在文档中的当前位置；每个光标须标明所属用户；用户移动或修改文本时，光标位置须**实时**更新。

#### 文档状态

每份文档须有**当前状态**，反映其生命周期阶段；状态决定用户可执行的操作及文档在各功能区的可见性。

系统须提供以下状态：

| 状态 | 说明 |
|------|------|
| **草稿** | 文档已创建，仅对所有者与已获查看/编辑/批注权限的用户可见 |
| **审批中** | 已对该文档启动审批；按既定顺序对指定的审批人可见 |
| **已批准** | 审批流程已成功完成，视为正式批准 |
| **已驳回** | 被审批流程中任一方驳回；此类文档**不允许**再修改 |

- 文档内容**仅**在「草稿」状态可编辑。
- **仅**在「草稿」与「审批中」状态允许新增批注。
- 已批准文档对**全体系统用户**可查看。

#### 审批流程

- 系统须允许用户对所选文档**启动审批**。创建流程时，用户确定审批人名单与审批类型。
- **无严格顺序**审批时，全体指定审批人可独立查看文档并作出审批决定。
- **有指定顺序**时，审批人依次进行；每名后续参与者仅在前一名完成审批后方可访问文档。
- 须展示审批进度（例如：已审批人数 / 应审批总人数）。
- 指定审批人可访问文档**当前版本**、批注与**变更历史**。

#### 交办文档

- 系统须提供单独模块，列出**交办给当前用户审批**的文档。用户可看到待其决策的文档及当前审批状态。
- 用户可批准或驳回。驳回时须填写**驳回原因**，记入审批历史，全体流程参与者可查看。
- 已驳回文档**不得**再编辑或再批注。

#### 文档版本与重新审批

- 若至少一名审批人驳回，审批流程视为**未通过**。流程发起人可修改文档并**创建新版本**，新版本须**重新走完全部**审批流程。旧版本保留在系统中可查看，但在当前审批语境下视为**非当前**。
- 创建新版本后，作者重新启动审批。系统须允许审批人通过**比对上一版与新版**审阅变更；比对时须**可视化**展示差异，包括新增、修改与删除的文本片段；比对结果以文本形式展示并**高亮**变更。
- 当全体指定审批人按所选顺序类型**全部**完成审批时，文档视为已批准。文档状态须**无歧义**地反映当前情况，并在所有相关界面可见。

#### 客户附加期望（非强制）

实现下列功能**非强制性**，但在系统评定时将视为**加分项**：

- 希望在文档内容编辑区域支持**图片与表格**的操作。

#### 须交付资料

向潜在客户移交系统时，须准备下列文档包：

| 交付项 | 说明 |
|--------|------|
| **部署说明** | 系统管理员安装系统时使用 |
| **技术文档** | 说明与软件架构图；便于快速理解项目的各类图与文档：模块位置与职责、运行规则；若存在数据库，须含结构示意；前后端结构与逻辑说明；与输入信息交互的算法等 |
| **用户说明** | 具体操作说明；不熟悉本程序的人员应能仅凭说明完成功能要求中所描述的结果 |
| **软件产品源代码** | |
| **项目演示文稿** | |
| **项目演示视频** | |

客户须收到**分发包**，其中包含所开发应用的安装包及安装、部署所需的必要组件。

### 模块 B：演示实施

产品开发完成后，须向潜在客户演示。除明显的资金安排外，客户将关注下列方面：

1. **业务模式 / 经济效果**：成本结构、费用与投资回报逻辑的说明；资产价值论证及其融入现有资源基础的方式。
2. **后续改进方向**：如何通过技术升级支撑产品资本化的**规模化战略规划**。
3. **开发技术说明**：使用的语言、框架、平台与架构思路；说明它们如何缩短实施周期、简化代码维护、提升方案质量并降低技术风险。
4. **产品演示**（含广告宣传元素的宣传片、配合功能说明的屏幕录制演示等）。

**演示 10 分钟，答疑 5 分钟。**
